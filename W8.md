## Part 1：Agent 管線的模型配置 — AgentOpt 論文

### 核心命題

- Columbia 大學 DAPLab 的 **AgentOpt** 論文用 9 個模型、81 種組合的大規模實驗
- 顛覆假設：「最強的模型放在最重要的位置，就會得到最好的結果」

### 用 Opus 當 Planner？

- 常見直覺：Opus 當 Planner，Sonnet 或 Haiku 當 Executor，分工明確、成本可控
- AgentOpt 數據（同一任務、同一管線架構）：
  - 🏆 Ministral 8B（Planner）+ Opus（Solver）= **74.27%** 準確率
  - ❌ Opus（Planner）+ 任何模型（Solver）= **31.71%** 準確率
- 差距 **42 個百分點**，贏的那組 Planner 是 Opus 價格的 **三十三分之一**

### 最反直覺的實驗結果

- HotpotQA（多跳問答）基準，9 個模型所有兩兩組合測試
- Ministral 8B 是 Mistral AI 的 8B 參數端側小模型，定價 $0.15/百萬 token
- 倒數 11 名全是 Opus 當 Planner 的組合

### 為什麼最強模型反而最差？

- **Opus 太強了，強到不願意委派任務**
- 論文團隊分析執行日誌：9 種 Opus 當 Planner 的組合中，有 **7 種** 出現了此問題

### 跨基準的系統性發現

| 基準 | 最佳結果 | 核心洞察 |
|---|---|---|
| **HotpotQA**（多跳問答） | Ministral 8B + Opus = 74.27% | Planner 需要「守規矩」而非「聰明」 |
| **MathQA**（數學推理） | Opus（Answerer）+ Haiku 4.5（Critic）= 98.84% | Critic 模型選擇影響極小，9 個 Critic 準確率差距僅 2.9 pp |
| **GPQA Diamond**（研究生科學問答） | Opus 單獨 = 74.75% | 唯一「用最強模型就對了」的基準；純推理任務，能力排名可預測表現 |
| **BFCL v3**（多輪函數呼叫） | 三個模型並列 70% | 成本差 32 倍；同分時，選便宜的就對了 |

### 「直接用最好的模型」到底多浪費？

- 準確率相近情況下，最佳與最差模型組合成本差距：
  - HotpotQA：**21×**
  - MathQA：**118×**（最極端）
  - BFCL：**32×**
- 每天跑幾千次 Agent pipeline，選錯組合一個月就是幾萬美金的浪費

### 為什麼不能靠直覺選模型組合？

- 論文測試 8 種搜索演算法
- **LM Proposal**（GPT-4.1 推薦最佳組合）結果令人沮喪：
  - GPQA：表現尚可（直覺有效）
  - HotpotQA：僅 34%，遠低於最佳的 74.27%
  - BFCL：僅 45%
- Hill Climbing 在 HotpotQA 上也只有 52% 機率找到最佳組合
- 🏆 **贏家：Arm Elimination** — 基於信賴區間的逐輪淘汰策略，節省 40–60% 搜索預算
- 核心結論：Pipeline 中的角色交互是非線性的，不能從單一 benchmark 推導 pipeline 角色表現

### Anthropic 自己也在修正這個思路 — Advisor Tool

- **過去做法：** Opus 當主腦（Planner），便宜模型當手腳（Executor）
- **Advisor Tool 做法：** Haiku / Sonnet 跑主迴圈當執行者，Opus 退居幕後當「顧問」，只在關鍵時刻被呼叫
- TestingCatalog 測試：Haiku 搭配 Opus 顧問的表現大幅超過 Haiku 單獨運行，總成本仍低於直接用 Sonnet
- 容錯設計：若顧問呼叫失敗，執行者不會中止

### 任務類型決定最佳分工

| 任務類型 | 最佳策略 |
|---|---|
| 需要工具使用（如 HotpotQA） | 弱模型當 Planner 可能更好（不會「搶活幹」） |
| 純推理（如 GPQA） | 強模型獨挑就行 |
| 需要批判審查（如 MathQA） | Critic 角色的模型選擇影響不大，不值得花大錢 |
| 多步驟長程（如 SWE-bench） | 便宜模型跑主迴圈 + 強模型按需諮詢 |

### 你現在可以做什麼？

1. **審計你的 Agent 管線成本** — 列出每個角色用的模型與每月 token 消耗；pipeline 裡可能有「假高價值」角色
2. **警惕「能力過剩」的陷阱** — 檢查有沒有角色的模型「太聰明」，繞過了它該遵守的流程
3. **用系統化方法找最佳組合** — AgentOpt 開源，支援 LangGraph、AutoGen、CrewAI，核心 Arm Elimination 演算法可節省 40–60% 搜索預算
4. **別相信直覺，包括 AI 的直覺** — GPT-4.1 推薦的組合只有 34% 準確率，比最佳組合差一倍

### 一句話總結

> Agent 管線優化的單位不是「單一模型的能力」，而是「模型組合在特定任務上的匹配度」。用錯組合，你可能花 32 倍的錢，拿到一半的效果。

### 常見問題 Q&A

- **Q：小模型永遠比大模型適合當 Planner？** — 不是。GPQA 上 Opus 獨挑就是最佳。結論是「沒有普適的最佳組合」。
- **Q：只有 2 個角色的 pipeline，值得用 AgentOpt 嗎？** — 9 個候選模型 × 2 角色 = 81 種組合，暴力搜索 $50–120；Arm Elimination 可降到 $30–70。月 API 成本超 $500 就值得。
- **Q：這跟 LLM Router（如 RouteLLM）有什麼不同？** — Router 針對單個查詢做路由；AgentOpt 針對整個 pipeline 做組合優化。Router 省單次呼叫的錢，AgentOpt 省整條管線的錢。

## Part 2：Cost Stack 框架 — 12 個機制 × 6 個成本層

### 為什麼需要 Cost Stack 框架？

學生對 token 成本最常見的三種誤解：

- ❌ 誤解一：「Token 經濟學就是用 prompt 叫模型講話精簡一點」
- ❌ 誤解二：「省 token 就是 prompt engineering 的副產品」
- ❌ 誤解三：「Cache 跟 batch 就是全部的解法」
- ✅ 正解：Token 成本來自 **6 個不同的層次**，每層都有截然不同的 trade-off 邏輯與工程代價

### Cost Stack 總覽

> Money flows top-down，optimization happens at L4 — 越往下的層次，工程師的干預空間越大，但也越需要嚴謹的 eval 支撐。

| 層級 | 名稱 | 說明 |
|---|---|---|
| **L5** | 產品／使用者層（Product / User） | 產品定價、freemium、用量上限、SLA 設計（不在本課範圍，但要記住它存在） |
| **L4** | 應用層（Application）⭐主場 | 工程師 90% 的優化機會在這裡；分 Generation / Context / Architecture 三大子區 |
| **L3** | API 定價層（API Pricing） | Provider 提供的 7 大契約機制（每個都是一個 trade） |
| **L2** | 推論計算層（Inference Compute） | Provider 內部：speculative decoding (2–3×)、PagedAttention、continuous batching |
| **L1** | 訓練計算層（Training Compute） | Capital expenditure（$10M–$1B+，對 app dev 完全不可見） |
| **L0** | 訓練資料層（Training Data）🆕 2026 | Cloudflare Pay Per Crawl — 從免費變成有 price tag = Token 經濟學的新上游 |

### L4 應用層三大子區詳細機制

**Generation 子區：**
- caveman — output 軟壓縮（~3×）
- structured-cot — thinking 硬壓縮（22–43×）
- reasoning effort — provider 官方 thinking knob

**Context 子區：**
- semantic caching — vector similarity（68% hit）
- context compaction — server-side 自動 summarize（~60%）
- RAG vs long context — 1,250× cost gap

**Architecture 子區：**
- MCP / tool use overhead（134K tokens 警告）
- multi-agent cost explosion（15× tokens）
- LLM routing — inter-model dispatch（10–60×）

> 💡 學生應記得：**90% 優化在這層**

### 跨層機制

- **Self-hosting** 跨 L1–L3（自營模型 → L1/L2/L3 由你支付）
- **LLM FinOps / Observability** 跨所有層（量測你在哪一層花錢）

## § 1 — L0 層：訓練資料層｜Cloudflare Pay Per Crawl

### L0 的核心命題

- 前 11 個案例處理 token 已產生之後的事；Cloudflare 把上游一整層從不可觀測變成可觀測
- 首次在訓練資料流中引入**價格訊號**
- 關鍵數字：
  - **1B** 每日 HTTP 402 回應（全球爬蟲請求規模）
  - **$60M** Reddit ↔ OpenAI 實際成交年費
  - **73,000:1** 爬蟲：人類比（Anthropic 爬蟲 vs 真實訪客）
- 過去：爬蟲索引 → 搜尋 → 點擊，流量循環存在。現在：AI 直接消費內容，循環消失

### HTTP 402 = SDD 奇蹟示範

- 1996 年：HTTP 402 標記為 "Reserved for future use"
- 2025 年：AI 爬蟲經濟條件成熟，402 被啟用 — 等待了整整 **30 年**
- 學生最常的抗議：「現在不知道未來需求，寫 spec 沒用」
- 402 的反問：30 年前的 spec 設計者也不知道 AI 爬蟲會出現 — 但他們留了空間。好的 spec 是 **forward-compatible** 的

### 機制與依賴

- Cloudflare Pay Per Crawl 的運作建立在 **Verifiability** 之上：沒有 cryptographic bot identity，就無法計費
- 流程：AI 爬蟲請求 → WAF + Bot 身分驗證 → 規則判定或 402 → 付款處理與交付

## § 2 — L3 層：API 定價層｜Provider 7 大契約機制

### 7 大軸 — 每個都是一個 Trade

| 軸 | 你給出 | 你拿到 |
|---|---|---|
| **時間** | 容忍延遲 | -50% / +150% |
| **記憶** | Prefix 穩定 | Cache read -90% |
| **能力** | 接受較低能力模型 | 5–51× 價差 |
| **承諾** | 預付 / 包月 | Flat rate |
| **預測** | 預測一部分輸出 | 該段免費 |
| **格式** | 接受結構化輸出 | 間接省 retry |
| **客製** | Fine-tune 投資 | 長期單價降 |

### 時間軸：Batch / Flex / Standard / Priority

| Tier | 折扣/加價 | 延遲承諾 |
|---|---|---|
| Batch | -50% | 最多 24 小時 |
| Flex | 約 -50% | 即時但時段變動 |
| Standard | 0%（基準） | 即時 |
| Priority | +150%（2.5×） | 比 Standard 更快 |

- Priority 是反向 trade — 用金錢換速度

### 記憶軸：Prompt Caching

| Provider | Cache 寫入 | Cache 讀取 |
|---|---|---|
| Anthropic | 1.25×（5 min）/ 2×（1 hr） | 0.1×（-90%） |
| OpenAI | 無額外費用 | 0.5–0.1× |
| Google | 有寫入費用 | 大幅折扣 |

- Break-even：Anthropic 5-min cache 讀 1 次就划算；1-hour cache 讀 2 次以上才划算
- ⚠ Tool schema 改動 → invalidate 全部 cache（Tool 定義位於 cache hierarchy 最前端）

### 能力軸：Model Tier

| Provider | 同家族價差範圍 | 倍數 |
|---|---|---|
| Anthropic | Haiku 4.5 ($1/$5) → Opus 4.7 ($5/$25) | 5–25× |
| OpenAI | GPT-5 nano → GPT-5.5 | 51× |
| Google | Flash-Lite → Pro | ~10× |

- 隱性陷阱：Opus 4.7 採用新 tokenizer，對同樣文字多產生 **35% tokens**。每 token 單價沒漲，但 effective cost 已悄悄漲了 35%。**絕對不能只看標價**

### 其他 4 個契約機制

1. **Predicted Outputs** — OpenAI 獨有，你猜對的部分免費，用例為 code editing
2. **Structured Outputs** — 利用 special tokens 表達結構，避免 retry 浪費；function calling 也屬此類
3. **Provisioned Throughput** — Commit-based，從 per-call 計費升維到容量規劃
4. **Fine-tune / Distillation** — 長期單價降，需要 stable workload 與完整 eval pipeline 支撐

## § 3 — L4-Generation 層：caveman / structured-cot / reasoning effort

### 三案例光譜

| 機制 | 控制方 | 約束類型 | 壓縮倍數 |
|---|---|---|---|
| caveman | App dev | SKILL.md（軟） | ~3× |
| structured-cot | App dev | Grammar（硬） | 22–43× |
| reasoning effort | Provider | Enum knob | ~2–5× |

- 核心洞察：**spec formality 與壓縮倍數成正比**

### Case Study：caveman

- 核心哲學：「why use many token when few do trick」
- 用 SKILL.md 軟性指引模型產生極簡輸出
- 壓縮模式範例：
  - **正常：** "Your component re-renders because you create a new object reference each render. Wrap it in useMemo."
  - **full：** "New object ref each render. Inline object prop = new ref = re-render. Wrap in useMemo."
  - **ultra：** "Inline obj prop → new ref → re-render. useMemo."
- Output Token 縮減 **65%**，整體壓縮 **~3×**（軟性約束上限）

### Case Study：structured-cot

- 20 行 GBNF grammar 強制規範思考結構
- 達到**無訓練、無 fine-tune、無 reward model** 的推理壓縮
- Grammar 範例：
  ```
  root ::= think code
  think ::= "\n" "GOAL: " line "APPROACH: " line "EDGE: " line "\n\n"
  line ::= [^\n]+ "\n"
  code ::= [\x09\x0A\x0D\x20-\x7E]+
  ```
- 機制：**Inference time logit masking** — 每個 token 生成步驟，grammar 直接遮蔽不合法的 token
- 零訓練成本，純推理期介入

#### structured-cot 三組對照（最精彩的 ablation 設計）

| Mode | Mean Think Tokens | 相對壓縮 |
|---|---|---|
| FREE | 3,087 | 1× |
| PROMPT_TERSE（自然語言請求簡潔） | 2,298 | 1.34×（沒用） |
| FSM（grammar） | 138 | 22.4× |

- 結論：prompt 請模型簡潔 → 模型答應了沒做到。Grammar 強制執行 → 額外 **16.65×** 壓縮
- 這是 **formal spec 的執行力優勢**

#### structured-cot 的暗面：Reasoning Displacement

| 指標 | FREE | FSM_PLAN |
|---|---|---|
| Mean think tokens | 52 | 1,753（+33×） |

- 看 token 壓縮 — 43.3×
- 看 total tokens 壓縮 — **只有 5.0×**
- **Verifiability Mindset 教科書：** 作者主動找會打臉自己的 metric，並誠實呈現
- Grammar 約束壓縮了 thinking，模型把推理擠到 code comments 裡 — 表面數字亮麗，真實壓縮有限

### Case Study：Reasoning Effort

| Provider | 機制 | 預設值 |
|---|---|---|
| OpenAI | reasoning.effort = minimal/low/medium/high/xhigh | medium |
| Anthropic | thinking: {type: adaptive, effort: ...} | 須顯式啟用 |
| DeepSeek V4 | 唯一顯示真實 thinking tokens | — |

- 常見問題：Many developers enable extended thinking and never adjust the effort parameter, paying for thinking tokens on requests that don't benefit.

### structured-cot vs Reasoning Effort 對偶

| 維度 | structured-cot | Reasoning Effort |
|---|---|---|
| 控制方 | App developer | Provider |
| 約束 | Hard（grammar） | Soft（enum） |
| 需要 | Open-weights + llama.cpp | API + reasoning model |
| 壓縮倍數 | 22–43× | ~2–5× |
| Spec 形式 | Formal grammar | Discrete enum |

- 兩者可互補：API 場景用 Reasoning Effort 快速降成本，Open-weights 場景用 Grammar 追求極限壓縮

## § 4 — L4-Context 層：cache / compaction / RAG / MCP

### Case Study：Semantic Caching

- Prompt cache 抓「同一 prefix 重複」；Semantic cache 抓「不同問法但意思相同」
- 關鍵數字：
  - **-68.8%** API Calls 減少（GPT Semantic Cache, arXiv）
  - **67%** Hit Rate（VentureBeat 真實案例）
  - **-73%** Cost 縮減（VentureBeat 真實案例）
  - **12×** Hit Rate 提升（VectorQ vs static threshold）
- 主流工具：GPTCache、Bifrost、VectorQ

#### Semantic Cache 的 Verifiability 風險

- 流程：查詢嵌入 → 向量資料庫 cosine similarity（門檻 0.8） → 命中回 cached 回應 / 未命中呼叫 LLM
- 風險：語義相似但意圖不同的查詢可能被錯誤命中

### Case Study：Context Compaction

- Anthropic first-party feature，透過 API 參數觸發：
  ```
  context_management={"edits": [{"type": "compact_20260112"}]}
  ```
- 效果：
  - Customer service workflow：**-58.6% tokens**（122K → 86K tokens）
  - 一般 context 縮減：**60–80%**
  - Claude Code 預設：~95% 觸發率，保留 ~33K buffer

#### Compaction 的失敗模式

- ⚠ Compaction 會 summarize away CLAUDE.md 中的 critical rules，導致模型忘記重要約束
- Workaround 策略：
  - Critical rules 寫進 **system prompt**（不在 conversation history 內，不會被 compaction 影響）
  - 自訂 **compactPrompt**，明確保留檔案路徑、錯誤訊息、使用者糾正等關鍵資訊
- **Proactive context engineering 是工程，reactive /compact 是急救**

### Case Study：RAG vs Long Context 經濟學

| 指標 | RAG | Long Context（1M） |
|---|---|---|
| Per-query cost | $0.00008 | $2.00 |
| 成本倍數 | 1× | 25,000× |
| Latency | 1× | 30–60× 慢 |
| Mid-doc recall | 高 | ~60%（lost in the middle） |

- Elasticsearch Labs 實測：RAG 便宜 **1,250×** per query
- 注意：Anthropic 近期改變策略，Sonnet/Opus 4.6 提供 flat-rate 1M context，長 context 成本曲線正在快速變動

### 三條 Context 壓縮路徑彼此正交

| 路徑 | 攻擊目標 |
|---|---|
| caveman / structured-cot | Output / Thinking tokens |
| Context Compaction | Conversation context |
| RAG | Knowledge context |

- 三條路徑可疊加，理論上乘法收益。但每加一層，eval 負擔也疊加

### Case Study：MCP / Tool Use Hidden Cost

- 最容易被忽視的成本來源 — 對話還沒開始，token 已先耗盡：
  - **55K tokens**：5 個 MCP Server = 58 tools，光 schema 就佔 55,000 tokens
  - **17K tokens**：Jira MCP 單獨一個 server 的 overhead
  - **134K tokens**：極端案例，優化前的 overhead
- 對話還沒開始，context window 已被 tool schema 消耗殆盡

#### 對比：Traditional vs Code Execution

| 方法 | Token 消耗 |
|---|---|
| ❌ Traditional（1,000 records × 200 tokens/筆，逐筆帶完整 context） | 200,000 tokens |
| ✅ Code Execution（MCP Code Execution, subprocess + AST） | 1,200 tokens |

- 壓縮 **166×**

#### Anthropic 三個解法

1. **Token-efficient tool use beta** — Claude 4 預設啟用，output tokens 平均 -14%，最多 -70%
2. **Tool Search Tool** — 按需發現工具，不再 eager-load 所有 schema
3. **Programmatic Tool Calling** — 用 code execution 處理 intermediate results，避免逐筆傳送

### Skill vs MCP Agent — 架構選擇 = Token Cost 選擇

| 維度 | Skill | MCP |
|---|---|---|
| Load 時機 | Lazy（用到才讀 SKILL.md） | Eager（連接就 load schema） |
| Token cost | 低 | 高 |
| 適合 | 確定要用的能力 | 探索性、多元能力組合 |

- 加 MCP server 像加 npm package — 你以為免費，但每加一個吃 5–17K tokens
- **架構決策在第一行程式碼之前，就已決定了成本結構**

## § 5 — L4-Architecture 層：routing / multi-agent / self-hosting

### Case Study：LLM Routing

- 不壓縮 token，但讓每 token 變便宜 **10–60×**
- 核心思想：不同難度的任務分配不同能力的模型
- AWS 真實案例（每天 50,000 歷史題 + 50,000 數學題）：
  - 全部 Claude 3.5 Sonnet：月成本 **$14,850**
  - 路由分流後：月成本 **$8,044**（節省 46%）
  - Routing Overhead：月成本 **$108–189**（僅佔節省金額的 1.3%）

#### 三種主要 Routing 策略

| 策略 | Accuracy | Latency | Cost | 說明 |
|---|---|---|---|---|
| Static（hard-code） | 高 | 最低 | 最低 | 規則固定 |
| LLM-assisted | 高 | ~0.5s | 中 | 用 LLM 判斷 |
| Semantic | 中 | ~0.1s | 極低 | Embedding 相似度 |
| Hybrid | 高 | 中 | 中 | 混合策略 |

- 學術系統：LLMRouter（ulab-uiuc），16+ 演算法統一 benchmark
- ⚠ Router 失敗很安靜 — 仍然回答，只是貴或答得差。**必須量測 misroute rate**

### Case Study：Multi-Agent Cost Explosion

- Anthropic 親自承認的數字：
  - **1×**：一般 Chat（基準線）
  - **4×**：Single Agent + Tools
  - **7×**：Claude Code Agent Teams
  - **15×**：Multi-Agent System
- Token usage 解釋 **80%** performance variance

#### 多 Agent 不是免費的午餐

- Anthropic 官方文件兩段關鍵警語：
  - "Multi-agent systems are best suited for tasks where **result value far exceeds cost**."
  - "Domains that require all agents to share the same context or involve many dependencies between agents are **not a good fit** for multi-agent systems today."
- 這是 Anthropic 用自家系統測出來的實測結論

#### 適合 vs 不適合 Multi-Agent

| ✅ 適合 | ❌ 不適合 |
|---|---|
| Breadth-first research | 需要 shared context |
| Independent parallel directions | Agent 之間 dependencies 多 |
| Result value 高 | 線性任務 |
| 例：Claude Research、大型 codebase 平行探索 | 例：大部分 chat 應用 |

### Case Study：Self-Hosting Break-Even

| 來源 | Break-even 條件 | 月費用 |
|---|---|---|
| TokenMix | 月 API spend $20K | — |
| digitalapplied (code) | 600M tokens/月 | — |
| digitalapplied (chat) | 1.2B tokens/月 | — |
| Vucense (RTX 4090) | 硬體 8 個月回本 | $200/月 |

- 隱性殺手：**Engineer time = $750–3,000/月**，維運工程師的人力成本往往超過硬體本身

#### Self-Hosting 工具光譜

| 工具 | 定位 |
|---|---|
| **Ollama** | 本地開發、個人實驗首選，部署簡單 |
| **vLLM** | Production 首選，PagedAttention + continuous batching |
| **TensorRT-LLM** | NVIDIA 極致效能，針對 GPU 深度優化 |
| **SGLang** | MoE 模型友好，結構化生成支援佳 |

- NCKU lab 場景建議：混合架構 — self-host 處理預測 bulk，Claude API 處理 high-value bursty 需求

## § 6 — 跨層：LLM FinOps / Observability — 讓優化可被驗證

### 為什麼 Observability 是 Token 經濟學的前提

- Dashboard 只告訴你消耗了多少，**不告訴你哪個功能、哪個用戶、哪個 session 是主因**
- 沒有 observability → 前面 12 個優化都是盲目操作，無法知道是否生效，更無法發現 failure mode

### 工具光譜

| 工具 | 特色 |
|---|---|
| **Langfuse** | MIT 授權，Open-source，Self-hostable，最廣泛使用 |
| **Helicone** | Proxy-based，最快上手，一行程式碼接入 |
| **LangSmith** | LangChain 親生子，深度整合 |
| **LangWatch** | LLM-native eval 能力強 |
| **CloudZero / Finout** | Enterprise 等級，跨 provider 成本分配 |
| **Datadog LLM Monitoring** | 已有 Datadog 基礎建設的團隊最快擴展路徑 |

### 必量測的指標

1. **Cost per feature / user / session** — 最核心的成本歸因指標
2. **P50 / P95 Latency** — 長尾延遲往往反映隱性 retry 或 fallback
3. **Cache Hit Rate（分類）** — 區分 prompt cache vs semantic cache
4. **Tool Definition Overhead** — 每個 request 的 tool schema token 消耗
5. **Reasoning Token Consumption** — 檢查是否對不需推理的任務過度思考
6. **Error / Retry Rate** — Agent loop 失控的最早期信號

## § 7 — 機制疊加：乘法 ROI

### 機制可疊加 — 真實計算示範

| 步驟 | 機制 | 該層縮減 | 累積有效成本 |
|---|---|---|---|
| 0 | Baseline | — | 1.00× |
| 1 | Batch | × 0.5 | 0.50× |
| 2 | Prompt Cache（80% prefix） | × 0.28 | 0.14× |
| 3 | Routing Opus → Sonnet | × 0.6 | 0.084× |
| 4 | caveman 砍 50% output | × 0.7 | ~0.06× |
| 5 | structured-cot 砍 thinking | × 0.4 | ~0.024× |

- 理論上可達 **~40× 整體成本縮減**：從 $1,000/月 → $25/月

### 但每多一層機制…

- **工程複雜度上升** — 每個機制都需要額外整合與維護
- **Eval 負擔上升** — 每層機制都需要獨立的 eval pipeline 驗證
- **Failure Mode 數量上升** — 機制間可能互相干擾
- Token 經濟學的真正試煉不是「能不能省」，是「**能不能 verify 沒搞壞東西**」

### ROI 排序建議

| 優先順序 | 機制 | 說明 |
|---|---|---|
| ① | **Prompt Caching** | 最低投入、立即見效，幾乎零工程成本 |
| ②–③ | **Batch + Routing** | Batch 適合非即時 workload；Routing 在 workload 異質性高時 ROI 最大 |
| ④–⑤ | **caveman + Semantic Cache** | Output 壓縮與重複 query 攔截，需要基礎 eval pipeline |
| ⑥–⑧ | **structured-cot、Context compaction、RAG 改造** | 需要更成熟的 eval 能力 |
| ⑧ | **Self-Hosting** | 最大投入、最大潛在收益，需穩定 workload 與工程團隊能量 |

## § 9 — 課堂討論題

1. **Input cost 不確定性** — 2027 年做 AI 應用，最大不確定性是 input cost — 為什麼？怎麼設計實驗來 hedge 這個風險？
2. **三條戰線選一條** — 你的 workload 花到 4× 預算，Generation / Context / Architecture 三條戰線只能先做一條，先做哪條 ROI 最高？
3. **Multi-agent 值得的任務** — Multi-agent 15× cost — 什麼任務真的值得？舉一個 lab 例子並用數字證明 result value 超過成本。
4. **Hard vs Soft 疊加順序** — structured-cot（hard grammar）vs Reasoning Effort（soft enum）— 疊加時哪個先做？先做哪個邊際效益更高？
5. **Cloudflare 普及後的生態** — Pay Per Crawl 普及後，open-source LLM ecosystem 會發生什麼？對學術界的訓練資料取得有什麼長期影響？

## 收尾：課程總結

### 一句話總結

> **Token 經濟學的本質，是在每一層設計一份清醒的契約。**

- **每個機制都是一個 Trade** — 你讓出某些條件，換取成本優勢。沒有免費的壓縮，只有清醒的交換。
- **工程能力 = 看穿代價** — 在每一層清楚知道你在交換什麼，並選擇自己能承受的代價組合。
- **Verifiability 是底線** — 無法量測的優化不算優化。每個決策都需要 eval 支撐，才算完整。

### 第 12 個案例的特殊地位

- 前 11 個案例教你「**怎麼花得更少**」— Generation、Context、Architecture 各層的壓縮與優化技術
- 第 12 個（Cloudflare L0）教你「**這個錢是怎麼產生的、誰在收**」— 從訓練資料的上游開始理解 token 的生命週期與價值流動
- 少了任何一塊，學生對 Token 成本的理解就不完整。**L0 是整個 cost stack 的起點，也是最常被忽視的一層**
